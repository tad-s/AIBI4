■ 現在のデータの制約・課題
  ・注文日時は注文全体で1つのみ → 個別商品の注文タイミングが不明
  ・希望受取日時は事前予約リードタイムとしてしか使えない（滞留時間ではない）
  ・会員コード列はあるが0で埋まっているケースが多く、会員属性データが別管理
  ・客層（年齢・性別）・1組あたり客数・着席退席時刻の列が存在しない

  1. 分析① 客単価に影響を与える変数の特定（重回帰分析）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 分析目的
  客単価（1注文あたりの合計金額）を目的変数とし、
  さまざまな説明変数が客単価にどの程度影響しているかを重回帰で定量化する。

■ 現在のデータで対応可能な変数
  変数名          算出方法                          精度
  ─────────────────────────────────────────────────────
  曜日            注文日時.dt.dayofweek             ○
  注文時間帯      注文日時.dt.hour                  ○
  注文形態フラグ  店内飲食=1 / テイクアウト=0      ○（分類有り）
  注文品目数      商品明細の行数カウント            ○
  FD比率（近似）  ドリンク系キーワードで近似        △（キーワード依存で精度低）

■ 本番分析に追加で必要なデータ

  【不足データ①】客層情報
    必要列: customer_age_group, customer_gender, customer_segment
    形式例: 会員テーブルに age_group（10代/20代/30代/...）, gender（M/F/不明）
    取得元: 会員登録情報・アプリ会員データベース
    連携方法: 会員コードで LEFT JOIN
    補足: 非会員（ゲスト注文）は「不明」として扱うか、
          注文時の入力情報（人数選択UI等）から取得する

  【不足データ②】1組あたり客数（テーブル人数）
    必要列: party_size（INTEGER）
    形式例: 注文または伝票に people_count 列を追加
    取得元: 注文時の人数入力（店内飲食の人数選択UI）、
            または席・テーブル管理システム
    連携方法: 注文番号または伝票番号で結合
    補足: モバイルオーダーの場合、注文時に人数を入力させることで取得可能

  【不足データ③】正確なFD比率
    必要列: 商品マスタに menu_category（FOOD/DRINK/SIDE/DESSERT/OTHER）
    形式例: 商品マスタCSVに category 列を追加
    取得元: メニューマスタ（POS/Tablecheck/OES）
    連携方法: TOメニューID（商品明細TOメニューID）で商品マスタをJOIN
    補足: 現在はキーワードマッチで近似しているが、マスタがあれば正確に算出可能

■ 分析に必要な最低データ件数
  重回帰の説明変数が6〜8個の場合、統計的有意性を確保するため
  最低 300注文/月（理想: 1,000注文/月以上）

■ 期待される出力
  ・各変数の標準化係数（β係数）バーグラフ
  ・回帰モデルのR²（決定係数）
  ・各変数のp値（有意性検定）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  2. 分析② 客単価に影響を与えている商品の特定（重回帰分析）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 分析目的
  個々の商品を「注文したか/しなかったか（0/1）」の説明変数として重回帰し、
  客単価への正負の影響度を商品別に算出する。

■ 現在のデータで対応可能な内容
  ・注文番号単位で商品をワンホットエンコード → 重回帰 → 係数の正負で判断
  ・現状でも実行可能（既に実装済み）

■ 本番分析に追加で必要なデータ

  【不足データ①】商品カテゴリマスタ
    必要列: menu_id, menu_name, category（FOOD/DRINK/SIDE/DESSERT）,
            is_carb（炭水化物系フラグ）, is_alcohol（アルコールフラグ）,
            calorie（カロリー）, price_tax_in（税込価格）
    形式例:
      TOメニューID, 商品名, カテゴリ, 炭水化物フラグ, アルコール, カロリー
      2403071946301, 塩レモンホタテバーガー, FOOD, 0, 0, 520
      2306243918900, ドリップコーヒー（ICE）G, DRINK, 0, 0, 15
    取得元: POSシステムのメニューマスタ / OESメニューマスタ
    連携方法: TOメニューIDで JOIN

  【不足データ②】時系列商品マスタ（価格改定対応）
    補足: キャンペーン期間限定商品の場合、期間情報も含めること
    例: valid_from, valid_to 列を商品マスタに付与

■ 精度向上のための推奨事項
  ・上位商品だけでなく、全商品（ただし出現率5%以上）を対象に実施
  ・季節ごと（四半期）に係数を再計算してトレンドを把握する
  ・「炭水化物メニューは客単価と逆相関」などの仮説を季節・時間帯別に検証

■ 分析に必要な最低データ件数
  各商品が最低 50注文以上含まれる量のデータが必要
  （1か月分では不足することがある → 3〜6か月分を推奨）

  3. 分析③ 顧客グルーピング（ABC分析）+ グループ別商品・注文タイミング分析
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 分析目的
  客単価を高（A）・中（B）・低（C）にグルーピングし、
  グループ別の商品構成と、炭水化物系メニューの注文タイミングを比較する。

■ 現在のデータで対応可能な内容
  ・客単価によるH/M/Lグループ分け（実装済み）
  ・グループ別の商品出現頻度（実装済み）

■ 本番分析に追加で必要なデータ（特に「注文タイミング」分析用）

  【不足データ①】商品明細ごとの注文時刻
    現在の問題: 1注文内のすべての商品に同じ「注文日時」が付与されており
                「この商品を何番目に注文したか」が不明
    必要列: order_detail_timestamp（注文明細ごとの受注日時）または
            order_sequence（注文内での注文順番号 1,2,3...）
    形式例:
      注文番号, 伝票番号, TOメニューID, 注文時刻
      A001, V001, MENU_001, 2025-09-01 18:00
      A001, V001, MENU_002, 2025-09-01 18:35   ← 35分後に追加注文
      A001, V001, MENU_003, 2025-09-01 19:10   ← 追加注文
    取得元: テーブルオーダーシステム（TOシステム）のログ
            ハンディ端末の注文ログ
            セルフオーダー（テーブル端末）の注文ログ
    補足: モバイルオーダーの場合、追加注文が別注文番号で来ることがある
          その場合は会員コードや席番号で紐付けが必要

  【不足データ②】テーブル番号・着席情報
    必要列: table_id（席番号）, checkin_time（着席時刻）
    取得元: テーブル管理システム / 予約管理システム

  【不足データ③】「締め」メニューフラグ
    必要列: 商品マスタに is_締め（炭水化物フラグ）
    例: やきそば、ご飯もの、ラーメン、パスタ等に is_締め=1 を付与

■ 注文タイミング指標の算出方法（データが揃った場合）
  ① 最初の注文時刻を基準（0分）として各商品の注文時刻の差分を計算
  ② 30分ごとのブロック（1〜6等）に分割
  ③ グループ別・タイミングブロック別の商品注文割合を集計
  → 高単価グループ: 締め商品が後半に集中
  → 低単価グループ: 締め商品が前半から注文

  4. 分析④ マーケットバスケット分析（一緒に注文されるメニュー）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 分析目的
  同一注文（または同一テーブルの滞在中）に一緒に注文される商品の組み合わせを
  共起頻度・支持度・信頼度・リフト値で評価する。

■ 現在のデータで対応可能な内容
  ・注文番号単位での商品共起ヒートマップ（実装済み）
  ・上位ペアランキング（実装済み）

■ 本番分析に追加で必要なデータ

  【不足データ①】時系列の追加注文を同一セッションで統合する仕組み
    現在の問題: テイクアウトの場合、1回の注文のみ → 比較的完全
                店内飲食の場合、複数回の追加注文が別注文番号になる可能性
    必要な処理: 同一テーブル・同一着席時間内の注文をセッションとして紐付け
    取得元: テーブル管理システム（テーブルごとの注文履歴）

  【不足データ②】商品カテゴリ情報（精度向上のため）
    詳細: 分析② と同様の商品カテゴリマスタ
    活用: 「フードとドリンクの組み合わせ」「前菜とメインの組み合わせ」
          といったカテゴリ横断の共起分析が可能になる

■ アソシエーション分析の評価指標（実装推奨）
  ・支持度（Support）: 全注文中での組み合わせの出現割合
  ・信頼度（Confidence）: 商品Aを注文した場合に商品Bも注文する割合
  ・リフト値（Lift）: ランダム性を超えて一緒に注文される度合い
  ・リフト値 > 1.5 を「有意な組み合わせ」として抽出推奨

■ 分析に必要な最低データ件数
  mlxtend の Apriori アルゴリズム使用の場合、
  最低支持度 2% での分析には 5,000注文以上を推奨

  5. 分析⑤ ヘビー系 vs ライト系 食べ物と客単価傾向
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 分析目的
  脂っこい・ボリュームの大きい「ヘビー系」と軽い「ライト系」の
  注文客の客単価・注文点数・再来店率の傾向を比較する。

■ 現在のデータで対応可能な内容
  ・商品名キーワードによるヘビー/ライト分類（実装済み、精度は低）
  ・グループ別の平均客単価比較（実装済み）

■ 本番分析に追加で必要なデータ

  【不足データ①】商品属性マスタ（精度の高い分類のため）
    必要列:
      is_heavy（ヘビー系フラグ）: 大きいサイズ, 揚げ物, ダブルパティ等
      is_light（ライト系フラグ）: サラダ, グリル, 小さいサイズ等
      calorie（カロリー）: 数値で定義することで閾値分類が可能
      fat_g（脂質量g）: より客観的な「重さ」の定義に使用
      size_category（S/M/L/LL）: サイズ分類
    取得元: 商品マスタ（栄養成分表示データ / メニューエンジニアリング資料）

  【不足データ②】会員別の再来店頻度（リピーター分析との組み合わせ）
    必要列: 会員コードに紐付いた visit_count（累積来店回数）, last_visit_date
    取得元: 会員管理システム（CRM）
    活用: ヘビー系注文客がリピーターに多いか、ライト系客の方が来店頻度が高いか

■ 分類ルールの標準化（推奨）
  以下の基準で商品マスタに事前フラグを付与することを推奨:
    ヘビー系（is_heavy=1）: カロリー 700kcal 以上、または脂質 30g 以上
    ライト系（is_light=1）: カロリー 400kcal 以下、かつ脂質 15g 以下
    ミックス（both=0）: 上記に該当しない標準的なメニュー

  6. 分析⑥ 滞留時間別客単価（時間客単価算出）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 分析目的
  店内飲食における顧客の滞留時間（着席〜退席）を計測し、
  時間あたりの客単価（時間客単価 = 客単価 ÷ 滞留時間）を算出して
  座席回転率と売上効率のトレードオフを分析する。

■ 現在のデータで対応可能な内容
  ・テイクアウトの「注文日時〜希望受取日時」の差（リードタイム）のみ算出可
  ・店内飲食の滞留時間は完全に未取得

■ 本番分析に追加で必要なデータ

  【不足データ①】着席・退席時刻（最重要・必須）
    必要列: table_id（席番号）, checkin_time（着席時刻）, checkout_time（退席時刻）
    形式例:
      店舗コード, 伝票番号, table_id, checkin_time,         checkout_time
      91176,      000498,   A05,     2025-09-03 13:20:00,  2025-09-03 14:55:00
    取得元（優先順位順）:
      1. POSシステムのテーブルオープン/クローズログ
      2. セルフオーダー端末の会計ボタン押下時刻
      3. テーブル管理アプリ（TableCheck、Ebica等）のAPI
      4. レジ打ちの開始・終了タイムスタンプ

  【不足データ②】テーブル情報マスタ
    必要列: table_id, capacity（最大収容人数）, area（エリア区分：窓際/中央/カウンター等）
    取得元: 店舗設備マスタ

  【時間客単価の算出式】
    時間客単価（円/時間）= 合計金額(税込) ÷ 滞留時間(時間)
    例: 合計4,200円、滞留90分 → 4,200 ÷ 1.5 = 2,800円/時間

  【分析ができた場合の期待されるアクション】
    ・滞留時間が短い（〜30分）: 回転率高→座席効率が良い
    ・滞留時間が長い（120分超）: 客単価は高いが時間客単価は低い
    ・「時間客単価が低い長居グループ」への施策（ラストオーダー案内等）

  7. 全分析共通で必要な基盤データ・インフラ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

■ 共通マスタデータ

  【7-1】商品マスタ（最重要）
    ファイル名想定: menu_master.csv
    必要列:
      TOメニューID      : 既存データと結合するキー
      商品名            : 商品名と一致させること
      カテゴリ大         : FOOD / DRINK / SIDE / DESSERT / OTHER
      カテゴリ小         : バーガー / フライ / サラダ / コーヒー / 等
      炭水化物フラグ     : 0 or 1
      アルコールフラグ   : 0 or 1
      ヘビー系フラグ     : 0 or 1
      ライト系フラグ     : 0 or 1
      カロリー（kcal）   : 数値
