# AIBI4 V9 実装前確認メモ

作成日: 2026-06-29
対象: `etc/追加するベース分析の要件_20260629.pdf`

## 前提

V9はV8をベースに `v9/` として分離した。現時点ではV8コードのコピーであり、以下の追加分析はまだ実装していない。
理由は、PDF要件内で「追加実装する前に、まず要件内容に沿った出力ができるか／どのような内容で出力するか提示」とあるため。

## PDF要件の要約

KGI実現に向け、注文数を増やす「注文シナリオ（導線）」を設計するためのベース分析を追加する。
追加したい分析は以下6つ。

1. 注文の流れ
2. 売れるメニューの組み合わせ
3. 初期注文の影響
4. 注文の連鎖条件
5. 時間帯別の違い
6. 注文が止まるポイント

## 実現可能性

既存V8の取得データに以下が含まれていれば実現可能。

- `伝票番号` または注文単位ID
- `注文日時`
- `来店時間`
- `退店時間`
- `商品名`
- `数量`
- `単価` / `合計金額(税込)`
- `店舗名`
- `人数`

ただし、注文の「順序」や「追加注文」を扱う分析では、同一伝票内で `注文日時` が複数時点に分かれている必要がある。
もし全商品が同じ `注文日時` で返るデータの場合、注文遷移や追加注文の精度は下がるため、「同時注文バスケット分析」中心に切り替える。

## 追加ベース分析の出力案

### ⑦ 注文の流れ分析

目的: 来店後、どの商品カテゴリ・商品順で注文が進むかを可視化する。

出力:
- 注文ステップ別 Top 商品 / Top カテゴリ
- 1品目 → 2品目 → 3品目 の遷移ランキング
- 初期1品のみで止まる割合
- 2品目・3品目へ進む確率

グラフ案:
- 左: 注文ステップ別 Top 商品の横棒グラフ
- 右: 代表的な注文遷移 Top10

知見例:
- 初期注文で多い商品
- 2品目に進みやすい組み合わせ
- どのステップで離脱・停止が多いか

アドバイス例:
- 初期注文直後に提示すべき追加候補
- 1品で止まりやすい顧客への声かけ商品

### ⑧ 売れるメニューの組み合わせ分析

目的: 同時注文・連続注文の両面から、セット提案に使える組み合わせを抽出する。

出力:
- 同時注文ペア Top10
- 連続注文ペア Top10（Aの後にB）
- 支持度、同時注文率、遷移率
- ドリンク×フードのクロスカテゴリ組み合わせ

グラフ案:
- 同時注文ペア Top10 横棒
- 連続注文ペア Top10 横棒

知見例:
- セット化しやすい商品ペア
- 追加注文として自然に出やすい商品

アドバイス例:
- 卓上POP / レコメンド枠 / セットメニュー候補
- 店員声かけ・モバイルオーダー表示順への反映

### ⑨ 初期注文の影響分析

目的: 最初に何を注文すると、その後の注文数・客単価が伸びるかを把握する。

出力:
- 初期注文パターン別 平均注文点数
- 初期注文パターン別 平均客単価
- 初期注文パターン別 追加注文率
- 初期注文が1品のみ / 2品以上 / ドリンク＋フード の比較

グラフ案:
- 初期注文パターン別 平均注文点数・客単価の複合棒グラフ
- 追加注文率ランキング

知見例:
- 注文数を伸ばす初期構成
- 客単価を伸ばす初期構成

アドバイス例:
- 最初の注文でドリンク＋軽食をセット提示する
- 初期1品のみ客に対して早期追加提案する

### ⑩ 注文の連鎖条件分析

目的: 注文が続く条件を数量・時間・品数の観点で明確にする。

出力:
- 初期2品以上 vs 1品のみ の追加注文率比較
- 2回目注文ありの場合の3回目注文率
- 注文回数別の平均客単価
- 注文間隔別の次注文率

グラフ案:
- 注文回数別 ファネルチャート
- 条件別 追加注文率の棒グラフ

知見例:
- 追加注文が連鎖しやすい初期条件
- 何分以内の追加が次注文につながるか

アドバイス例:
- 最初の10分以内に追加提案する
- 2回目注文後に3回目を促す商品を提示する

### ⑪ 時間帯別の違い分析

目的: 時間帯ごとの注文行動の違いを明確にする。

出力:
- 時間帯別 平均注文点数
- 時間帯別 平均客単価
- 時間帯別 Top 商品 / Top カテゴリ
- 時間帯別 追加注文率

グラフ案:
- 時間帯別 平均注文点数・客単価
- 時間帯別 Top カテゴリの積み上げ棒グラフ

知見例:
- 軽い注文が多い時間帯
- 注文点数が伸びる時間帯
- 高単価商品が出やすい時間帯

アドバイス例:
- 18時台は軽いつまみ訴求
- 20時台は高単価・追加注文訴求
- カフェの場合はモーニング / ランチ / カフェ / ディナーの時間帯に合わせる

### ⑫ 注文が止まるポイント分析

目的: 追加注文が止まるタイミングと原因候補を可視化する。

出力:
- 何品目後に止まるかの分布
- 最終注文カテゴリ / 最終注文商品 Top10
- 注文間隔が長くなった後の追加率
- 滞在時間に対する最終注文時刻の位置

グラフ案:
- 最終注文ステップ分布
- 最終注文カテゴリ Top10
- 注文間隔別 追加率

知見例:
- 2品目後に止まりやすい
- 特定カテゴリで注文が終わりやすい
- 注文間隔が一定以上空くと追加率が低下する

アドバイス例:
- 注文停止前に提案すべき商品
- ラストオーダー前の声かけタイミング
- 滞在後半向けの締め・デザート・追加ドリンク提案

## 改善提案

PDFの6分析は注文導線分析として有効。ただし、既存のベース分析①〜⑥とは目的が異なるため、V9では以下のように分けるのがよい。

- ベース分析①〜⑥: 客単価・商品・顧客グループ・滞在効率の分析
- 注文導線分析⑦〜⑫: 注文数を増やすための行動シナリオ分析

UI上も「ベース分析」と「注文導線分析」をタブ分離する方が理解しやすい。

## 継続チャット対応案

現状V8も `chat_history` は保持しているが、プロンプトが「ユーザーに再確認を求めない」方針のため、曖昧な指示でも無理にグラフを作る。
V9では以下へ変更する。

### 方針

- 明確な分析指示: そのまま集計・グラフ生成
- 曖昧な分析指示: すぐグラフ化せず、確認質問を返す
- データ構造上不可能な指示: 不可能な理由と代替分析案を返す
- ユーザーが追加回答したら、過去の文脈を踏まえて分析を実行

### 具体例

ユーザー: 「新宿の売れ筋を見たい」

V9応答:
- 「新宿」は複数店舗候補があります。対象は以下のどれですか？
- 商品別売上、注文数、客単価貢献のどれで見ますか？

ユーザー: 「新宿南口、商品別売上で」

V9応答:
- 前回の意図を引き継ぎ、店舗=新宿南口、指標=商品別売上でグラフ生成

### 実装方針

- `chat_router.py` に会話状態 `pending_analysis` を保存
- `llm_service.py` のシステムプロンプトから「再確認禁止」を撤廃
- LLM応答を以下3タイプに分類
  - `answer_only`
  - `clarification_needed`
  - `analysis_with_graph`
- `clarification_needed` の場合はコード実行しない
- 次回入力時に `pending_analysis` と `chat_history` を統合して分析意図を補完

## 実装前の結論

要件内容に沿った出力は可能。
ただし注文順序系の分析は `注文日時` の粒度に依存するため、実装時には以下のフォールバックを入れる。

- 注文日時が複数時点に分かれている場合: 遷移・連鎖・停止分析を実施
- 注文日時が同一伝票内でほぼ同一の場合: 同時注文・初期構成・品数分布中心に切り替える

次の実装ステップは、V9に「注文導線分析⑦〜⑫」と「継続チャットの確認質問フロー」を追加すること。
